# Intent Agent（意图识别 Agent）

意图识别（Intent Classification）是对话系统和 Agent 的核心能力之一。它将用户的自然语言输入映射到预定义的动作类别，是理解用户"想做什么"的第一步。

## 为什么需要意图识别？

当用户说"帮我订一张明天从北京到上海的机票"时，系统需要理解：

1. **用户想做什么**（意图）→ 订机票（book_flight）
2. **关键信息是什么**（槽位）→ 出发地=北京，目的地=上海，日期=明天
3. **有多大把握**（置信度）→ 92% 确定是订机票

没有意图识别，Agent 无法将自然语言映射到具体的行为。意图识别是 Agent 决策链中的第一环。

## 核心概念

### 意图分类（Intent Classification）

意图分类将用户输入归类到预定义的意图类别中。每个意图代表一种用户期望的操作：

| 意图名称 | 中文标签 | 示例输入 |
|----------|---------|---------|
| `book_flight` | 订机票 | "帮我订一张去上海的机票" |
| `book_hotel` | 订酒店 | "预订后天杭州的酒店" |
| `query_weather` | 查天气 | "北京今天天气怎么样" |
| `cancel_order` | 取消订单 | "我要取消订单 A12345" |
| `play_music` | 播放音乐 | "放一首周杰伦的歌" |

意图分类的本质是一个**多分类问题**：给定输入文本，输出最匹配的意图类别和对应的置信度分数。

### 槽位填充（Slot Filling）

槽位（Slot）是执行意图所需的关键参数。不同的意图有不同的槽位定义：

**订机票意图的槽位：**

```
输入: "帮我订一张明天从北京到上海的机票"

提取结果:
┌──────────┬────────┬────────────┐
│ 槽位名   │ 原始值 │ 归一化值   │
├──────────┼────────┼────────────┤
│ origin   │ 北京   │            │
│ dest     │ 上海   │            │
│ date     │ 明天   │ 2026-03-23 │
└──────────┴────────┴────────────┘
```

**槽位归一化** 是将用户的口语化表达转换为标准格式：
- 日期："明天" → "2026-03-23"，"下周一" → "2026-03-24"
- 地点："帝都" → "北京"，"魔都" → "上海"
- 数量："两个人" → 2

**必填 vs 可选槽位：**
- **必填槽位**：缺少时 Agent 需要主动追问（如订机票必须有出发地和目的地）
- **可选槽位**：有默认值或可以省略（如舱位等级默认经济舱）

### 多意图处理（Multi-Intent）

用户有时会在一句话中表达多个意图：

```
"帮我订明天去上海的机票，顺便查一下上海的天气"
```

这句话包含两个意图：
1. `book_flight`（订机票）— 主意图，置信度较高
2. `query_weather`（查天气）— 次要意图

多意图处理策略：
- **优先级排序**：按置信度排序，优先处理最高置信度的意图
- **串行执行**：按顺序依次处理每个意图
- **并行执行**：同时处理多个独立的意图

### 置信度评估（Confidence）

置信度表示模型对分类结果的确信程度，通常是 0-1 之间的浮点数：

- **≥ 80%**（高置信度）：可以直接执行，无需确认
- **60% - 79%**（中等置信度）：建议确认，"您是想要订机票吗？"
- **< 60%**（低置信度）：需要用户进一步澄清

## 使用 LLM 实现意图识别

### 传统 NLU vs LLM

| 维度 | 传统 NLU | LLM 方案 |
|------|---------|---------|
| 训练数据 | 需要大量标注数据 | 零样本或少样本即可 |
| 新增意图 | 需要重新训练模型 | 修改 Prompt 即可 |
| 灵活性 | 只能识别训练过的意图 | 可以理解任意表达 |
| 推理速度 | 毫秒级 | 通常 1-3 秒 |
| 成本 | 一次性训练成本 | 每次调用都有 API 费用 |
| 适用场景 | 高吞吐、低延迟场景 | 灵活性优先的场景 |

### System Prompt 设计

使用 LLM 做意图识别的关键在于设计合适的 System Prompt：

```text

## 角色
你是一个专业的意图识别系统。给定用户的自然语言输入，你需要从预定义的意图列表中进行分类，并提取相关槽位信息。

## 预定义意图列表

${intentDescriptions}

## 任务要求

1. **意图分类**: 分析用户输入，判断其最匹配的意图，并为所有预定义意图给出置信度评分（0到1之间的小数，所有意图的置信度之和应大致为1）
2. **槽位提取**: 从用户输入中提取与主意图相关的槽位值
3. **槽位归一化**: 如果槽位值可以标准化（如相对日期"明天"、"后天"转为具体日期），在 normalized 字段中提供标准化值。今天的日期是 ${new Date().toISOString().split('T')[0]}
4. **推理说明**: 简要说明你的分类推理过程

## 输出格式

请严格按照以下 JSON 格式返回结果，不要包含其他内容：

{
  "primaryIntent": {
    "name": "意图名称",
    "label": "意图中文标签",
    "confidence": 0.92
  },
  "slots": [
    {
      "name": "槽位名称",
      "label": "槽位中文标签",
      "value": "原始提取值",
      "normalized": "归一化值（如无则省略此字段）"
    }
  ],
  "allIntents": [
    {
      "name": "意图名称",
      "label": "意图中文标签",
      "confidence": 0.92
    }
  ],
  "reasoning": "推理过程说明"
}

## 注意事项
- allIntents 数组应包含所有预定义意图，按置信度降序排列
- 置信度应该是合理的，最匹配的意图置信度最高
- 如果用户输入不太匹配任何预定义意图，所有意图的置信度都应较低
- slots 只提取与主意图相关的槽位
- 只返回 JSON，不要有其他文字
```

**设计要点：**

1. **角色设定**：明确告诉 LLM 它是一个意图识别系统
2. **意图列表**：提供完整的意图定义和槽位说明
3. **输出格式**：使用 JSON Schema 约束输出结构
4. **归一化指令**：明确要求对日期等值进行标准化
5. **评分指引**：说明置信度的含义和分配规则

### JSON Structured Output

通过要求 LLM 输出严格的 JSON 格式，可以实现结构化的意图分析结果：

```json
{

    "name": "book_flight",
    "label": "订机票",
    "confidence": 0.92

  "slots": [

```

## 最佳实践

1. **意图设计原则**
   - 意图之间应互斥，避免语义重叠
   - 每个意图的范围要清晰，不宜过大或过小
   - 意图命名使用 `snake_case`，语义明确

2. **槽位设计原则**
   - 区分必填和可选槽位
   - 为每个槽位提供类型约束
   - 考虑归一化需求（日期、地点、数量等）

3. **置信度阈值**
   - 设定合理的阈值决定是否需要用户确认
   - 所有意图置信度都低于阈值时，应主动追问

4. **错误处理**
   - 用户输入与所有预定义意图都不匹配时，应优雅降级
   - 槽位提取不完整时，提示用户补充信息
   - LLM 返回格式异常时，做好容错处理

5. **Prompt 优化**
   - 提供清晰的意图描述和示例
   - 使用 Few-shot 样例提高准确率
   - 控制输出温度（temperature 设置为较低值如 0.1）
